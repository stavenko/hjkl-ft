# Распознавание еды по фото («по фото еды»)

**Статус:** проектный документ перед реализацией. Код ещё не пишем — зафиксированы
ТЗ, факты из исследования и **принятые решения** (§10, дефолты обоснованы
исследованием — можно переопределить). Числа и модели проверены по публичным
источникам (ссылки внизу).

---

## 1. Постановка (ТЗ)

В форме продукта (`FoodEditor`) сейчас две вкладки: «По описанию» (текстовый
lookup) и «По фото» (OCR этикетки). Меняем:

1. Вкладку **«По фото» → «По этикетке продукта»** (смысл не меняется — это OCR
   этикетки, один продукт, КБЖУ на 100 г).
2. Добавляем **третью вкладку «По фото еды»** — распознавание блюда/тарелки:
   - VLM (vision-нейросеть) перечисляет **всю видимую еду с оценкой массы**:
     ```
     [огурец] 200 г
     [чечевица] 150 г
     [тыквенные семечки] 30 г
     ```
   - Для каждой позиции ищем **нечёткое соответствие** среди продуктов
     пользователя в IndexedDB (тоже через нейросеть — точного совпадения имён
     не будет).
   - Для позиций, которых у пользователя нет, — наш **стандартный lookup по
     описанию** (`ai::lookup`), который отдаёт КБЖУ.
   - Детектируем **скрытые жиры**: у салатов без видимого масла добавляем
     подсолнечное масло как долю массы; у салатов с белой заливкой
     (майонез/сметана/йогурт) — майонез как долю массы.
   - Весь список выводим на форму: каждая позиция = **название + граммы +
     наши стандартные КБЖУ-бейджи**. После сборки всего списка разблокируем
     кнопку **«Добавить все продукты»**.

**Глобальные ограничения проекта, которые здесь критичны:**
- Скрытые жиры (масло/майонез) — это **гипотеза, а не измерение**: показываем
  их **явно, отдельными позициями, с возможностью убрать**. Не «запекать»
  молча в итог (CLAUDE.md: не додумывать молча; гипотезы помечать).
- Оценка массы по одному 2D-фото неточная (см. §6) — всегда **редактируемая**.

## 2. Целевой UX-флоу

1. Пользователь добавляет 1+ фото еды (переиспользуем HEIC→JPEG кодек и
   фото-грид из вкладки этикетки).
2. Жмёт «Определить еду» → on-prem Qwen2.5-VL возвращает список `{name, grams,
   confidence}` (имена — русские).
3. Фоновая пост-обработка (по каждой позиции, с ограничением конкурентности):
   - коррекция массы (bias больших порций, см. §6);
   - matching к продуктам пользователя → `food_id | null`;
   - для `null` — `ai::lookup(name)` → `product_name` + КБЖУ;
   - hidden-fat проход → добавляет позиции «масло»/«майонез» (флаг «inferred»).
4. Список рендерится: `[название] [_____ г] [🔥ккал][Б][Ж][У]`, каждую позицию
   можно править (граммы/название) и удалять; inferred-позиции визуально
   помечены.
5. Все позиции готовы → «Добавить все продукты» → bulk-запись в дневник
   (переиспределяем `save_food_to_diary` / draft→food→diary).

## 5. Ответ №2 — Модели для нечёткого сопоставления (matching)

Задача: «дано детектированное имя X и список N продуктов пользователя — вернуть
лучший `id` или `null`». Два подхода:

**A. Эмбеддинги (рекомендуется как основной):**
- `@cf/baai/bge-m3` — **мультиязычный** (русский!), 1024-мерные векторы, 60K
  контекст, GA, дёшево (~1 075 нейронов/1M токенов). Эмбеддим список продуктов
  один раз, косинус-сравниваем каждую детекцию. Детерминированно и дёшево.

**B. Текстовый LLM (tie-breaker / fallback):**
- `@cf/meta/llama-3.1-8b-instruct` — дёшево, приличный русский, надёжный JSON
  при узкой схеме; ~8× дешевле 70B на выходных токенах. Эскалация на
  `@cf/meta/llama-3.3-70b-instruct-fp8-fast` только на неоднозначных синонимах.
- Альтернатива без нового биндинга: **переиспользовать наш `ai::lookup`-стек на
  `@cf/qwen/qwen3-30b-a3b-fp8`** для matching-промпта (см. §6.2).

**Рекомендованная связка:** bge-m3 (retrieval top-K) → LLM только для
disambiguation «трудных» near-ties. При маленьком списке продуктов (обычный
пользователь — десятки-сотни) можно начать **сразу с LLM** и без эмбеддингов,
а bge-m3 добавить, если список большой.

**Решение D2 (см. §10). Фаза 1 — чистый LLM-matching** (порог по confidence),
через существующий ai-стек, без нового биндинга и векторного кэша. bge-m3 —
если упрёмся в цену/скорость на больших списках.

## 6. Ответ №3 (часть A) — Алгоритмы детекции и оценка массы

**Реалистичная точность (из статей/бенчмарков):**
- Общие VLM (GPT-4V/Claude) по массе: **MAPE ≈ 36–37%**; ошибка по порции
  GPT-4V ~54.6 г против ~43.6 г у диетологов (люди тоже плохо оценивают порции,
  ошибка ~40–53%).
- **Систематическое занижение больших порций** — главный failure mode
  (маленькие ~точно, большие занижаются в 1.3–1.5×). → добавляем **восходящую
  коррекцию, растущую с размером** («для чужой готовки прибавь треть»).
- Топ-1 распознавание блюда в реальном мире ~**60–90%** (зависит от кухни).
- Depth/reference-object методы дают ~2–5% ошибки, но требуют глубины/эталона —
  у нас один RGB-кадр без пропсов, поэтому не применимо.

**Что реально помогает (закладываем в промпт):**
- **Эталонные объекты в кадре**: тарелка (обеденная ~26 см, десертная ~20 см),
  вилка (~19 см), рука, монета. Просим модель **назвать, какой эталон
  использовала**.
- **Плотность/стандартная порция** для перевода объём→граммы; при отсутствии
  эталона — «стандартная порция» + поправка на долю покрытия тарелки.

**Вывод для дизайна:** ожидаем **~35–40% ошибки массы на позицию** и занижение
крупных порций. Приемлемо для дневника при условии: (а) масса редактируемая,
(б) коррекция bias больших порций, (в) эксплуатация видимого эталона.

### 6.1 Промпт итемайзера (пример)

```
Ты — nutrition vision assistant. Из фото выведи ТОЛЬКО ту еду, которую реально
ВИДНО. Не выдумывай вероятные гарниры.

Шаги (рассуждай про себя, потом выдай JSON):
1. Каждую отдельную видимую еду — отдельной позицией. Для салата/боула перечисли
   различимые компоненты (зелень, белок, семечки…), но НИКОГДА не считай одну и
   ту же еду дважды.
2. Порция: найди эталон масштаба (край тарелки ~26см обед / ~20см десерт,
   вилка ~19см, рука, монета). Назови использованный эталон. Оцени граммы из
   покрытия тарелки + типичной плотности. Нет эталона — «стандартная порция»,
   так и укажи.
3. confidence 0–1 на позицию.

Строгий JSON, без прозы:
{"items":[{"name":"огурец","grams":200,"cue":"plate_coverage",
"confidence":0.7}, ...], "notes":"..."}

Правила:
- Имена — на РУССКОМ, короткое каноническое название продукта.
- Сомневаешься, есть ли предмет — ПРОПУСТИ (лучше не досчитать, чем выдумать).
- Не объединяй две еды в одну и не дроби одну на две.
- grams — одно число, не диапазон.
- Большие тарелки часто НЕДООцениваются — не занижай крупные порции.
```

### 6.2 Промпт matching (пример)

```
Сопоставь ДЕТЕКТИРОВАННУЮ еду со списком продуктов пользователя.
Сопоставляй по СМЫСЛУ (сорт/готовка/форма слова могут отличаться), а не по
точному написанию.

Детектировано: "тыквенные семечки"
Кандидаты (id, name):
1: семечки подсолнечника
2: семена тыквы жареные
3: кешью

Строгий JSON:
{"match_id": <id или null>, "confidence": <0-1>, "reason": "<кратко>"}

Правила:
- Матч только если это ТА ЖЕ еда (сорт/готовка могут чуть отличаться).
- Другая еда, лишь похожая по звучанию, -> match_id: null.
- Несколько подходят -> ближайший + ниже confidence.
```
Здесь `тыквенные семечки` → id 2 (`семена тыквы жареные`). Одноязычное
сопоставление по смыслу — сильная сторона LLM, точного совпадения имён не надо.

Пороги: `confidence ≥ 0.8` — авто-матч; `0.5–0.8` — «предложено, подтвердите»;
`< 0.5` или `null` — уходим в `ai::lookup` (внешний КБЖУ).

## 7. Ответ №3 (часть B) — Скрытые жиры: масло и майонез

Опубликованного «алгоритма» нет — это **prior, добавляемый поверх видимой
массы**, флагом и с возможностью отклонить. Опорные числа (USDA):

- Оливковое/подсолнечное масло: **884 ккал/100 г**, 1 ст.л. ≈ 13.5 г ≈ **120 ккал**.
- Майонез (обычный): **~680 ккал/100 г**, 1 ст.л. ≈ 14 г ≈ **~94 ккал**.
- Сметана: ~198 ккал/100 г.
- Стандартная порция заправки: 2 ст.л. (~30 г) на ~2 чашки зелени (~85 г) —
  это ~35% массы зелени. Масло в oil-and-vinegar салате: ~5–12% массы.

### Рекомендованные допуски (по умолчанию)

**Направление ошибки.** Консенсус трекеров (Cronometer/MFP) — **слегка
ЗАВЫШАТЬ** скрытый жир (его чаще всего забывают, он калориен). Формулировка
«чтобы не обманывать» допускает оба чтения; но недосчёт скрытого жира в
диет-приложении срывает дефицит. **Решение D3 (см. §10): реалистичные допуски,
каждая inferred-позиция явная и удаляемая** — тогда «обмана» нет. Ниже —
реалистичный и консервативный столбцы для справки.

| Скрытый жир | Триггер | Консерв. дефолт (не завышать) | Реалистичный | Ккал-эффект | Источник/логика |
|---|---|---|---|---|---|
| Масло в салате | салат без видимой заправки | **5%** масла (≈1 ч.л./100 г) | 10% | +~45 ккал/100 г | низ диапазона 5–12% |
| Белая заправка (майонез/сметана/йогурт) | видна белая заправка | **10%** (≈2 ч.л./100 г), считаем как майонез | 20% | +~65–70 ккал/100 г | ~⅓ от порции 30 г |
| Масло в жареных/тушёных овощах | готовые овощи, масло не видно | **5%** масла (≈1 ч.л./100 г) | 8–10% | +~45 ккал/100 г | низ диапазона впитывания |

**Trade-off (зафиксировать):** консервативные 5–10% **ниже** норм диетологов
(15–35% заправки, до 25% впитывания при жарке) — защищают от завышения, но
**недосчитают** обильно заправленные ресторанные салаты и фритюр. Рекомендация
для re:Norma (диет-приложение, где недосчёт скрытого жира — типичная причина
«не сходится дефицит»): взять **реалистичный** столбец, но каждую inferred-
позицию делать **явной и удаляемой** — тогда «обмана» нет, пользователь видит и
контролирует. Финальный выбор — за тобой.

## 8. Цены и лимиты Cloudflare Workers AI

- Единица — «нейроны»; переплата **$0.011 / 1000 нейронов**. Бесплатно —
  **10 000 нейронов/сутки**.
- Нейронов на 1M токенов: `llama-3.2-11b-vision` 4 410 in / 61 493 out;
  `llama-3.1-8b` 25 608 / 75 147; `llama-3.3-70b-fp8-fast` 26 668 / 204 805;
  `bge-m3` 1 075 (in).
- Rate-limit текстовой генерации ~**300 req/min** (vision под тем же лимитом).
- **Vision доминирует по цене** — ресайзим картинки, ограничиваем `max_tokens`.
  Бесплатных 10k нейронов/сутки хватит на скромное число vision-вызовов —
  для прода нужен Workers Paid.

## 10. Принятые решения (по итогам исследования)

Дефолты выбраны на основе §4–§8 и текущей архитектуры. Пользователь может
переопределить, но по умолчанию строим так:

- **D1 — Vision: on-prem Qwen2.5-VL — та же модель, что уже крутит `ocr-queue`.**
  Причина — **оптимизация одного сервера**: on-prem бокс один, а
  перезагрузка/передёргивание моделей стоит существенного времени, поэтому
  держим на нём **ровно одну** VL-модель. `ocr-queue` уже гоняет Qwen2.5-VL для
  OCR этикеток → новую «еду по фото» вешаем на **тот же** инстанс: добавляем в
  очередь тип задания **«food-items»** (мульти-итем промпт §6.1 + схема
  `AiFoodItemsOutput`), без второй модели и без CF-vision. Qwen2.5-VL — топ
  открытых VL и силён в русском, так что качество здесь лучшее из доступного
  без второго сервера.
  - **📌 TODO (будущее): апгрейд on-prem до Qwen3-VL-8B** (почти drop-in, влезает
    в 24 ГБ; OCR 32 языка, лучше grounding/порции, 8B обгоняет прошлый 72B).
    Грабли: `transformers ≥ 4.57`, свежий vLLM, **patch size 14** от 2.5-VL
    нужно сменить (иначе токены молча рассинхронятся). Делаем **не сейчас** —
    сначала фича на 2.5-VL, апгрейд отдельным заходом.
  - CF-vision (`mistral-small-3.1-24b` / `llama-3.2-11b-vision`) и сторонний API
    (OpenRouter/DashScope) — **отвергнуты**: CF-модели заметно слабее Qwen, а
    сторонний API = egress фото пользователя на чужой хост. Оставлены в §4 как
    справка.
- **D-bridge — язык: НЕ нужен.** Qwen2.5-VL силён в русском → перечисляет еду
  **сразу русскими названиями**, matching к продуктам пользователя одноязычный
  (проще и надёжнее). Английский детур из прошлой (CF-)редакции убран. Для
  непойманных `ai::lookup` как обычно отдаёт русский `product_name` + КБЖУ.
- **D2 — Matching: LLM-matcher (фаза 1).** Промпт §6.2 через существующий
  ai-стек (`@cf/qwen/qwen3-30b-a3b-fp8`); без нового биндинга и векторного кэша.
  У обычного пользователя список продуктов — десятки/сотни, LLM тянет. bge-m3
  эмбеддинги вводим только если упрёмся в цену/скорость на больших списках.
- **D3 — Допуски масла/майонеза: реалистичные (масло 10%, белая заливка 20%,
  жарёные овощи 8–10%), каждая inferred-позиция явная и удаляемая.** re:Norma —
  диет-приложение, где недосчёт скрытого жира срывает дефицит (типичная причина
  «не сходится»); реалистичные нормы диетологов ближе к правде. «Обмана» нет,
  т.к. масло/майонез показаны отдельными строками с пометкой «добавлено
  автоматически» и убираются одним тапом (CLAUDE.md: гипотеза помечена явно).
- **D4 — Обработка позиций с ограничением конкурентности** (как `classify`).
  CF Workers AI держит rate-limit ~300 req/min на текст/vision — не шлём весь
  список залпом, идём небольшими батчами/последовательно.
- **D5 — Матчи по confidence:** `≥0.8` авто-применяем; `0.5–0.8` показываем как
  «предложено» (продукт подставлен, но визуально помечен — пользователь видит и
  может сменить на lookup); `<0.5`/`null` → `ai::lookup`. Без модального
  опроса на каждую позицию — всё правится прямо в списке.
- **D6 — Bulk-add создаёт отдельный `Food` + отдельную diary-запись на каждую
  позицию** (как обычный лог), а не один «составной» продукт. Переиспользуем
  `save_food_to_diary` / draft-механику; фоновая классификация тегов — как есть.
- **D7 — Лимиты входа:** до N фото и ресайз ~768–1024px на разбор (бюджет
  латентности/нейронов). Конкретное N подберём на первой итерации (старт: 3).
