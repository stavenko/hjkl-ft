# Возобновление подписки

Путь «назад» после [отмены](cancellation.md) или истечения. Здесь два разных механизма:
автоматическое рекуррентное продление (пока не отменено) и повторная подписка (после того,
как доступ пропал).

---

## 1. Автоматическое рекуррентное продление (happy)

Пока подписка не отменена, lava списывает деньги каждый период и шлёт webhook.

- **lava webhook `subscription.recurring.payment.success` → `POST /webhook/lava`**:
  1. `parse_webhook` → `Recurring`; `contractId` (+ `parentContractId` для дочернего инвойса), `periodEnd`.
  2. Резолв `user_id`: `PaymentIndexDO` `contract:<contractId>` (и `contract:<parentContractId>`) → `user_id`.
     - На первом платеже payment-worker также кладёт `contract:<root>` → `user_id`, чтобы рекуррент находил аккаунт.
  3. **`SubscriptionDO(user_id)./activate`** `{periodEnd, activateKey}`: продлевает `end` до
     нового `periodEnd`. Идемпотентно по `activate_key` — повтор webhook `end` вперёд не двигает.
  4. Push «Подписка продлена» (main-flow).
- **Модель данных:** меняется только `SubscriptionDO.end` (вперёд, к `periodEnd` провайдера).
  Ничего не создаётся. **Внешние:** lava webhook (входящий).

## 2. Повторная подписка после истечения

Доступ пропал (`end < now` → гейт `Locked`; либо после [возврата](refund.md) `expired`).
Приложение направляет оформление в **Telegram Mini App** (не на сайт).

- Пользователь снова проходит [покупку](onboarding.md) в Mini App → **новый** claim
  (новый `claim_id`, новый lava-контракт) → оплата → webhook `paid`.
- Mini App `/miniapp/me` по новому `paid`-claim → «Получить доступ» → `/onboard#claim=…`.

### ⚠ Пробел: привязка новой оплаты к СУЩЕСТВУЮЩЕМУ аккаунту

Онбординг-страница (`OnboardPage`) на `#claim` всегда идёт по ветке **регистрации** (шаг
7.1 happy-path — создать аккаунт + passkey), а `/claim` привязывает к только что созданному
`user_id`. Для **уже зарегистрированного** пользователя это означает:

- по ссылке `/onboard#claim` он создаст **новый аккаунт**, и подписка привяжется к нему, а
  не к старому (со всей его историей/данными);
- «чистого» пути «войти существующим аккаунтом → привязать новый платёж» в онбординг-UI нет
  (`/claim` — authed-эндпоинт и технически это поддерживает: залогиниться passkey → получить
  JWT → `POST /claim` привяжет к существующему `user_id`; но UI такой сценарий не разводит).

**Следствие для модели данных:** повторная оплата существующим пользователем сейчас
безопасно ложится только на НОВЫЙ аккаунт. Чтобы возобновлять на старый аккаунт, нужен UX
«войти → привязать claim» (логин passkey, затем `/claim` под этим JWT). — открытый вопрос.

## 3. «Отменил, но передумал» (до истечения)

Подписка отменена (`no_renew=true`), но ещё активна. Эндпоинта «снять отмену / возобновить
автопродление» **нет** — контракт в lava уже удалён (`DELETE /subscriptions`). Возврат к
автопродлению = оформить подписку заново (см. п.2) после истечения текущего доступа. Именно
поэтому в Mini App/приложении для отменённой подписки показывается «возобновление возможно
через N дней — когда истечёт доступ».

## Сводка

| Механизм | Триггер | Меняет | Внешние |
|----------|---------|--------|---------|
| Рекуррент | webhook lava | `SubscriptionDO.end` вперёд | lava webhook |
| Повторная подписка | пользователь в Mini App | новый `ClaimDO` + (при онбординге) новый аккаунт | lava invoice + webhook |
| Снятие отмены | — | нет механизма | — |

> Терминологическая правка: **«повторная оплата» — не отдельный спроектированный
> сценарий.** Спроектированы только (а) первая покупка и (б) авто-рекуррент. Ручная вторая
> оплата — emergent-путь: страница подписки уводит истёкшего пользователя в Mini App, а
> гостевой чекаут на каждый вызов мятит новый claim, ничего не зная о прошлых покупках.

## 🔧 Найденные проблемы и решения

### F-2: чекаут не идемпотентен, `payUrl` не хранится («guest» — не совсем guest)
**Проблема.** `miniapp_checkout` на каждый тап «Оплатить» зовёт `call_internal_checkout` →
мятит **новый claim + новый инвойс lava**. `payUrl` **нигде не сохраняется** → кто создал
ссылку, но не оплатил, при повторном открытии видит форму снова и создаёт **второй инвойс**.
При этом личность у нас есть — `tg_user_id` лежит в `TgSessionDO.miniapp_claims` («guest»
только по отношению к app-аккаунту).

**Решение — идемпотентность по `tg_user_id`:** перед созданием проверять claim'ы пользователя:
- **pending** (ссылка есть, не оплачена) → вернуть **ту же `payUrl`** (для этого её надо
  **хранить** — на `ClaimDO` или в `miniapp_claims`), новый инвойс не плодить;
- **paid** (не привязан) → не платить → «Получить доступ» (онбординг);
- **claimed active** → уже подписан.

**Затрагивает:** `telegram-worker` (`miniapp_checkout`/`miniapp_me`), хранение `payUrl`.

### F-3: повторная оплата привязывается к НОВОМУ аккаунту
**Проблема.** См. выше: `OnboardPage` всегда регистрирует новый аккаунт → повторная оплата
существующего пользователя ложится на пустой аккаунт, старая история остаётся Locked. Нет
UX «войти → привязать новый claim к существующему аккаунту» (хотя `/claim` authed это умеет).

**Решение:** развести онбординг на две ветки — «новый пользователь» (register + claim) и
«уже есть аккаунт» (login passkey → `/claim` под существующим JWT). Данные готовы.

### F-4: нет «снять отмену» (un-cancel)
Контракт в lava после `DELETE /subscriptions` удалён; вернуть автопродление можно только
новой подпиской. Возможно, стоит явно это коммуницировать (уже частично: «возобновление
через N дней»).

**Статус F-2…F-4:** зафиксировано, не реализовано.
