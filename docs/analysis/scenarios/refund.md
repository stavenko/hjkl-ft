# Возврат средств

Ветвление от [отмены](cancellation.md): у пользователя **отменённая, но ещё активная**
подписка (оплаченные дни не использованы), и он просит вернуть деньги за остаток. В
отличие от отмены — доступ обрывается **мгновенно**.

**Точка входа:** PWA → Управление подпиской → «Запросить возврат» (доступна только когда
`no_renew = true` и подписка ещё активна). Сам возврат денег — **вручную оператором в
дашборде lava** (API возврата у lava нет).

---

## Поток по шагам

### 1. Диалог с расчётом суммы
- `POST /refund/preview` (payment-worker, JWT) → `compute_refund(user_id)`:
  1. `SubscriptionDO` → `end`, `contract_id`. Нет активной → `no_active_subscription` (400).
  2. **Цена = фактический последний платёж из lava** (с учётом промокода):
     `Lava::last_payment(contract_id)` → **lava `GET /api/v2/invoices`** → самый свежий
     `COMPLETED`-инвойс по контракту (или его рекуррентным детям) → `receipt.amount`.
     - **Fallback'а на цену из конфига НЕТ.** Нет контракта / нет платежа / lava недоступна
       → **500 + `console_error`** (возвращать реальные деньги от прайс-листа нельзя).
  3. `days_left = ceil((end − now) / день)`.
  4. **`amount = round(price × 0.92 / 30 × days_left)`** (−8% комиссия, дневная ставка × остаток).
  5. Возвращает `{amount, currency, daysLeft}` — **без побочных эффектов**.
- Диалог показывает: предупреждение, что доступ прервётся сразу; сумму; примечание, что на
  обработку нужна неделя + время банковского возврата. Кнопка подтверждения заблокирована,
  пока сумма не загрузилась (при 500 — понятная ошибка, подтвердить нельзя).

### 2. Подтверждение → `POST /refund/request`
- `compute_refund` пересчитывается (авторитетно), затем:
  1. **`ClaimDO./refund-add`** — заявка оператору: строка `refunds`
     `{user_id (PK), amount, currency, contract_id, email, days_left, created_at, status:"requested"}`.
     Одна открытая заявка на пользователя (повтор перезаписывает).
  2. **`SubscriptionDO./refund`** — `end = now`, `status = "expired"` → **доступ отзывается
     мгновенно** (гейт `active = now < end` сразу становится false).
  - Возвращает `{ok, amount, currency}`.
- Порядок: сперва записываем заявку (чтобы оператор увидел), затем отзываем доступ.

### 3. После запроса
- **PWA**: статус перечитывается → `expired` → гейт `Locked`. Доступ пропал.
- **Оператор**: `GET /admin/refunds` (экран «Платежи» в консоли) — заявка с суммой, email,
  `lava contract`, остатком дней. Оператор идёт в **дашборд lava** и оформляет возврат
  вручную (минус ~8% комиссия lava; API нет).

## Модель данных

| Хранилище | Что происходит | Источник |
|-----------|----------------|----------|
| `ClaimDO.refunds` | **создаётся** строка заявки `{amount, currency, contract_id, email, days_left, status:"requested"}` | compute_refund + sub (email/contract) |
| `SubscriptionDO(user_id)` | `end→now`, `status→"expired"` (мгновенный отзыв) | — |
| lava | только читается (`GET /api/v2/invoices`) для суммы | реальный `receipt.amount` |

**Деньги — только из lava:** сумма считается от фактического платежа (промокод учтён); при
недоступности lava — жёсткий отказ (500), а не догадка по прайс-листу. Реальное движение
денег происходит вне системы — вручную в lava.

## Отражение возврата обратно в системе

lava НЕ шлёт webhook о возврате (события возврата в API нет). Когда оператор оформит
возврат/void контракта в lava, это отразится через сверку unbound-платежей
(`subscriptionDetails.terminatedAt`/`CANCELLED` в `GET /api/v2/invoices`) → авто-`void`
соответствующего claim. Для уже-`expired` подписки дополнительного эффекта нет.
