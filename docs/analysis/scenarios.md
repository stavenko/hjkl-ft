# Сценарии re:Norma — деньги

Анализ построен вокруг **одного счастливого пути** — онбординг человека на PWA с
лендинга с оформлением подписки — и его **ветвлений**, которые приводят к выходу
человека (прерванный онбординг, отмена, возврат) или к возвращению (возобновление).

Пока охвачены только сценарии, связанные с деньгами. Остальные домены (auth-в-чистом-виде,
sync, поддержка/чат, OCR, push, cron, баг-репорты) — отложены.

## Счастливый путь

- **[Онбординг с лендинга с подпиской](scenarios/onboarding.md)** — лендинг → Telegram
  Mini App → оплата (lava) → возврат в Mini App → регистрация (passkey) → привязка
  подписки → вход в PWA.

## Ветвления (выход человека)

- **[Прерванный / незавершённый онбординг](scenarios/onboarding-interrupted.md)** — где
  цепочка может оборваться: оплатил, но не забрал; passkey не сработал; claim не прошёл;
  закрыл PWA на полпути. Что остаётся в данных и как это видит оператор.
- **[Отмена подписки](scenarios/cancellation.md)** — стоп автопродления, доступ до конца
  оплаченного периода; отражение в боте и Mini App.
- **[Возврат средств](scenarios/refund.md)** — pro-rata по фактическому платежу, мгновенный
  отзыв доступа, заявка оператору (сам возврат — вручную в lava).

## Возвращение

- **[Возобновление подписки](scenarios/renewal.md)** — после отмены/истечения человек
  оформляет подписку заново (через Telegram Mini App); плюс автоматическое рекуррентное
  продление до отмены.

## Находки

- **[Открытые проблемы (F-1…F-4)](findings.md)** — не-атомарный онбординг, не-идемпотентный
  чекаут, привязка повторной оплаты к новому аккаунту, отсутствие un-cancel.

## Durable Objects, задействованные в деньгах

| DO | Воркер | Роль в денежных сценариях |
|----|--------|---------------------------|
| `TgSessionDO` | telegram-worker | `miniapp_claims`: claimId → {tg_user_id, secret} для Mini App |
| `ClaimDO` | payment-worker | реестр гостевых оплат (pending→paid→claimed→void) + `refunds` |
| `PaymentIndexDO` | payment-worker | `claim-contract:<order>` → claimId (маршрутизация webhook) |
| `SubscriptionDO` | payment-worker | подписка аккаунта (status/end/no_renew); источник гейта |
| `AuthDO`, `UserDO` | auth-worker | аккаунт + ключ входа (создаётся на шаге регистрации) |

Полный разбор полей каждого DO — в [durable-objects.md](durable-objects.md).

## Внешние API в деньгах

- **lava.top** (`gate.lava.top`, `X-Api-Key`): `POST /api/v3/invoice` (чекаут),
  webhook `payment.success` / `subscription.recurring.payment.success` /
  `subscription.cancelled`, `DELETE /api/v1/subscriptions` (отмена),
  `GET /api/v2/invoices` (фактическая сумма для возврата / сверка). **API возврата у
  lava нет** — возврат оформляется вручную в дашборде.
- **Telegram Bot API**: сообщения пользователю (оплата прошла / подписка отменена).
- **WebAuthn** (браузер): регистрация и вход по passkey.
